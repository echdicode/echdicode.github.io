Dependency inversion principle(DI principle)?

Trước khi đi vào giải thích thì các bạn hãy xem ví dụ sau:

class DaddyA{}
class DaddyB{}
class DaddyC{}
class sugarBaby{
  private DaddyA daddyA;
  public sugarBaby(){
    daddyA=new DaddyA();
}
}

Như các bạn đã biết sugarBaby mỗi tháng cần DaddyA chu cấp và bao bọc để có một cuộc sống mang hơi ấm “gia đình”. Lúc này DaddyA sẽ là dependency (phụ thuộc) của sugarBaby.

Và đã là sugarBaby chuyên nghiệp rồi thì việc thay đổi nhiều daddy thay đổi không khí “gia đình” là chuyện thường xuyên(bạn nào làm sugarbaby rồi thì confirm chỗ này giúp mình nhé). Nhưng như code trên các bạn có thể thấy sự phụ thuộc giữa sugarBaby và DaddyA là rất mạnh và việc thay đổi rất khó khăn. Và từ đây Dependency inversion principle ra đời để giải quyết cho vấn đề này :>.

Dependency inversion principle có nghĩa là các class cấp cao không nên phụ thuộc vào các class thấp và cả hai nên phụ thuộc vào abstraction(trong OOP thường là dùng interface).

Nghe có vẻ abstraction đúng không vậy chúng ta hãy trở lại với câu chuyện của cô sugarBaby kia nhé:

public interface SugarDaddy{ }
public class DaddyA implements Sugar Daddy {}
public class DaddyB implements Sugar Daddy {}
public class Daddyc implements Sugar Daddy {}
class SugarBaby {
  private Sugar Daddy daddy;
  public SugarBaby (Sugar Daddy daddy) {
    this. daddy = daddy;
  }
}
public static void main(String[] args) {
  DaddyA daddyA = new DaddyA();
  DaddyA daddyB = new DaddyB();
  DaddyA daddyC = new Daddyc();

  SugarBaby baby = new SugarBaby (daddyB);
}

Ok bây giờ SugarBaby và các daddy đều đã phụ thuộc vào interface SugarDaddy lúc này thì sự phụ thuộc giữa SugarBaby và DaddyA đã lỏng lẻo hơn nhiều. SugarBaby có thể nhận các daddy khác một cách dễ dàng chỉ cần tạo một daddy và implements SugarDaddy và Inject là ok.
